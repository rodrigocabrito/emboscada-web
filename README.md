# Emboscada — Plataforma da Equipa

Internal web platform for **Emboscada Parque Aventura** (a paintball / laser tag / adventure park). It's a staff-only tool for managing game sessions, the team, monitor scheduling, ammunition stock, evaluations, and internal announcements.

> This entire project — application code, serverless functions, security rules, tests, and this README — was generated by AI (Claude), through iterative prompting. It is provided as-is.

## What it does

- **Authentication & roles** — email/password login (Firebase Auth) with three staff roles:
  - **Administrador** — full access to everything.
  - **Monitor Líder** — can edit session line items (packs/extras/others) and register payments; can be scheduled like a monitor.
  - **Monitor** — can edit session line items; session data is view-only.
- **Início (Dashboard)** — each user's upcoming assigned sessions at a glance.
- **Sessões** — calendar view (day/week), full session detail (client info, players, type, caliber, packs/extras/others, live financial summary and payment flow).
- **Disponibilidades** — monthly availability calendar (full / partial with time limits), including Portuguese public holidays.
- **Equipa** — team directory grouped by role, with tenure achievements.
- **Comunicados** — internal announcements with emoji reactions; admins publish/edit, everyone reacts.
- **Perfil** — personal info, password change (forced on first login), and the user's own evaluation summary.
- **Admin panel**
  - **Utilizadores** — create users (auto-generated password + welcome email), promote/demote monitors ↔ monitor leaders, delete users.
  - **Sessões** — filterable/sortable table of all sessions.
  - **Catálogo** — manage packs, activities and prices.
  - **Stock** — ammunition stock per caliber (.50 / .68) with restocks, manual counts, removals and per-session usage tracking.
  - **Escala** — assign monitors to a day's sessions based on their availability, with scheduling-rule warnings.
- **Email notifications** — welcome email on user creation (CC to admins), notification to all users when a comunicado is published, and a notification to a user when their evaluation is updated.

## Tech stack

- **Frontend:** React 19, Vite, React Router, TanStack Query
- **Backend-as-a-service:** Firebase (Authentication + Cloud Firestore)
- **Serverless:** Vercel Functions (`/api`) for privileged/admin actions — Firebase ID tokens verified with `jose`, admin actions performed with a Firebase service account via Google REST APIs, email delivered through Gmail SMTP (`nodemailer`)
- **Testing:** Vitest (unit tests for the pure business logic)
- **CI:** GitHub Actions (lint + test + build on every push/PR)
- **Hosting:** Vercel

## Running locally

### Prerequisites
- **Node.js 20+** and npm
- A **Firebase** project with Authentication (Email/Password) and Cloud Firestore enabled

### 1. Install
```bash
npm install
```

### 2. Configure environment
Create a `.env.local` file in the project root with your Firebase web-app config (see [Environment variables](#environment-variables)):

```bash
VITE_FIREBASE_API_KEY=...
VITE_FIREBASE_AUTH_DOMAIN=...
VITE_FIREBASE_PROJECT_ID=...
VITE_FIREBASE_STORAGE_BUCKET=...
VITE_FIREBASE_MESSAGING_SENDER_ID=...
VITE_FIREBASE_APP_ID=...
```

### 3. Start the dev server
```bash
npm run dev
```
The app runs at the URL Vite prints (default `http://localhost:5173`).

> **Note:** the `/api` serverless functions (email, user management, announcements) do **not** run under the plain `vite` dev server. Use `vercel dev` locally, or a Vercel deployment (preview or production), to exercise them.

## Serverless functions (`/api`)

All three verify the caller is an authenticated **admin** before doing anything, so the endpoints can't be abused.

| Endpoint | What it does |
|----------|--------------|
| `POST /api/send-email` | Sends a branded email to given recipients (admin-triggered). |
| `POST /api/admin-users` | Creates a user (Firebase Auth account + Firestore profile + welcome email with a random password) or deletes one (profile + Auth account). Runs server-side so the admin isn't signed out and credentials never reach the browser. |
| `POST /api/publish-announcement` | Creates the announcement document **and** emails the team in one request, so the notification can't be lost if the tab closes. |

Sensitive writes (creating users, publishing announcements) happen here via the service account rather than the client, which is why the corresponding Firestore rules block those operations from the browser.

## Security rules

Firestore security rules live in [`firestore.rules`](./firestore.rules). Publish them via **Firebase Console → Firestore Database → Rules → paste → Publish** (or `firebase deploy --only firestore:rules` if you set up the Firebase CLI).

They require an authenticated staff account for all access, enforce field-level session permissions per role, restrict sensitive collections to admins, and block client-side creation of users/announcements (those go through the serverless functions).

> ⚠️ **If you change `firestore.rules`, you must re-publish them** — they are not deployed automatically with the app.
